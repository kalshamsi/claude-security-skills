# v1.4.0 CSO Audit — Skill Description Review

## Purpose

Audit all 10 skill descriptions against Anthropic's writing-skills best practices
and propose rewrites for v1.5.0. No SKILL.md edits land in v1.4.0. This memo is
purely diagnostic: it identifies the class of CSO violation present in the
current descriptions and stages concrete rewrites so the v1.5.0 re-run of the
100 behavioral prompts can measure whether CSO compliance moves the triggering
score.

## Audit rubric (from writing-skills)

Authoritative source: `~/.claude/skills/writing-skills/SKILL.md` (the
`writing-skills` skill). Rules below are taken verbatim from its "Claude Search
Optimization (CSO)" section, not invented.

- **Description = triggering conditions, NOT what the skill does.** The
  description exists so Claude can decide "should I read this skill right now?"
  at load time. Triggers belong in the description; capabilities belong in the
  body.
- **Start with "Use when..."** to focus on symptoms, situations, and contexts.
  Technology-specific skills should make that explicit in the trigger itself
  (e.g., "Use when auditing Python code").
- **Third person, injected into the system prompt.** No "I can help you...".
- **Size budget.** `<500` chars preferred, `<1024` chars hard cap on the whole
  frontmatter. Name must be letters/numbers/hyphens only.
- **Do NOT summarize the skill's process or workflow.** Quoting writing-skills
  directly: *"Testing revealed that when a description summarizes the skill's
  workflow, Claude may follow the description instead of reading the full skill
  content... The trap: descriptions that summarize workflow create a shortcut
  Claude will take. The skill body becomes documentation Claude skips."* This
  is the specific failure mode this audit targets — see the hypothesis section.
- **Keyword coverage.** Include the actual error messages, symptoms, synonyms,
  and tool names Claude would search for. Triggers should describe the
  *problem* (race condition, weak encryption) not the language-specific symptom
  (`setTimeout`, `Fernet`) unless the skill itself is technology-specific.
- **Active, verb-first naming.** Already satisfied across the suite (all 10
  names are hyphenated noun phrases or gerund-led).

The pattern this audit flags is a specific subset of rule 1/5: every
description in the current suite **leads with a capability summary** ("X
scanning via Y", "Generate Z") and then appends the "Use when..." triggers as a
second sentence. Per writing-skills, that leading clause is exactly the
workflow summary Claude will shortcut to.

## Candidate root-cause hypothesis

The v1.3.0 test report (`docs/test-report.md`) documented a triggering-average
of **2.0/3.0** across the suite, with three skills scoring triggering=1 on
keyword prompts: bandit-sast (T-1: "scan Python for security vulnerabilities"
did not trigger), crypto-audit (T-1: "weak encryption and hardcoded
cryptographic keys" did not trigger despite being an exact keyword match), and
docker-scout-scanner (T-1: skill was "found" but the workflow was skipped). The
report explicitly notes for crypto-audit: *"Keyword triggering (T-1) failed —
same pattern as bandit-sast. The skill's description alone is insufficient for
automatic activation without the explicit slash command."*

Per writing-skills (`~/.claude/skills/writing-skills/SKILL.md`, CSO section),
this is the documented failure mode of descriptions that mix workflow summary
with triggering conditions: *"Testing revealed that when a description
summarizes the skill's workflow, Claude may follow the description instead of
reading the full skill content... The skill body becomes documentation Claude
skips."* Every description in this suite leads with a capability summary (e.g.,
"Python SAST scanning via Bandit.") before the "Use when..." triggers. That
leading clause is exactly the shortcut writing-skills identifies. It plausibly
explains why the skill appears to "not activate" on keyword prompts even when
the trigger vocabulary is a near-exact match: the agent is reading the leading
summary, treating it as sufficient context, and proceeding without loading the
SKILL.md body that contains the workflow, Findings Format, and install/fallback
ladder.

v1.5.0 should apply the proposed rewrites in the per-skill table above and
re-run the same 100 prompts from `docs/test-report.md`. Expected: triggering
average rises from **2.0** toward **3.0**; boundary average (currently
**1.3**) stays approximately unchanged because boundary is a different failure
mode driven by the "When NOT to Use" section in the SKILL.md body, not the
frontmatter description. If triggering moves and boundary doesn't, the
hypothesis holds. If neither moves, the root cause is elsewhere (e.g.,
skill-loading behavior in the harness) and the v1.5.0 investigation needs to
pivot.

## What we are NOT doing in v1.4.0

- **No SKILL.md edits.** The 10 rewrites in the per-skill table above are
  drafts, not commits. They are staged here so v1.5.0 has a concrete starting
  point.
- **No behavioral re-run.** The 100 prompts from `docs/test-prompts.md` stay
  frozen. v1.4.0 ships the measurement infrastructure (CI, structural tests,
  methodology doc) without running the experiment.
- **No badge claims about triggering.** The README will not claim an improved
  triggering score in v1.4.0. Any "10/10 passing" or "triggering avg X.X" badge
  waits for v1.5.0, after the re-run produces fresh numbers.
- **No prompt revisions.** The 100 prompts are the experimental control. They
  do not move between v1.3.0 and v1.5.0 so the before/after delta is
  attributable to the description rewrites alone.
